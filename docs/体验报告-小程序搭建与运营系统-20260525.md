# 小程序搭建与运营系统体验报告

**测试日期**：2026-05-25  
**测试环境**：Windows 本地环境，后端 `http://127.0.0.1:8080`，后台 `http://127.0.0.1:3001`，微信开发者工具服务端口 `127.0.0.1:56647`  
**测试账号**：后台管理员 `admin/admin123`  
**测试角色**：产品测试经理、运营人员、小程序用户  
**测试结论**：有条件通过。后台发布数据、小程序公开页面 DSL、小程序运行态渲染已经形成闭环；但公开配置接口权限、小程序真实登录能力、可视化截图回归和部分导航落点仍存在风险。

---

## 一、执行摘要

本次重点复测用户前期反馈的“后台搭建页面与微信开发者工具编译页面不一致”“页面内容没有同步”“开发者工具出现未知数据源和 unknown component”等问题。

从当前证据看，首页已经不再停留在旧的异常态：公开页面 DSL 返回的组件类型为 `banner, nav, member_card, article_list, product_list`，微信开发者工具运行态也能读取相同组件类型，小程序 WXML 中能看到真实运营内容，例如“五一活动限时优惠”“金卡会员”“积分”“优惠券”等。

整体判断：核心首页发布同步问题已经明显改善，后台发布版本与小程序运行态基本一致；但系统还不能直接判定为完全验收通过，因为配置公开接口返回 403、微信登录曾出现 `invalid appid`，且当前测试未覆盖支付、下单、预约等需要真实微信身份或第三方能力的链路。

## 三、产品测试经理视角

### 1. 核心链路验证

本次按“后台发布数据 -> 公开 API -> 微信开发者工具运行态 -> WXML 渲染结果”的链路验证。

通过链路：

1. 后台管理员可登录，说明管理端认证链路可用。
2. 版本管理可读取最新发布版本，当前最新为 `1.2.0`，状态为已发布。
3. 公开首页 DSL 能返回发布后的核心组件，不再出现空组件或仍指向旧模板的问题。
4. 微信开发者工具运行态能读取同一组组件类型，页面状态为非 loading、无运行态错误。
5. WXML 中出现实际运营文案和会员数据，说明渲染层已经拿到内容并完成组件输出。

当前未完全闭环的链路：

1. `/api/v1/mp/config/public` 返回 403，公开配置接口权限策略需要确认。小程序运行态虽然能获得部分全局配置，但公开配置端点本身不可直接访问，会影响后续独立小程序端、联调工具或第三方检测。
2. 微信登录链路在用户前期开发者工具日志中出现 `POST /api/v1/mp/auth/login 422` 与 `invalid appid`。本轮没有重跑真实微信登录，因为这依赖有效 AppID 和微信开发者工具项目配置。
3. 支付、报名、预约、领券、商品详情等深层业务链路未完成真实端到端验证。

### 2. 缺陷与风险分级

| 编号 | 等级 | 问题 | 影响 | 建议 |
|---|---|---|---|---|
| QA-01 | P1 | `/api/v1/mp/config/public` 返回 403 | 公开配置接口不可独立访问，可能导致小程序配置、主题、tabbar 在某些环境无法加载 | 明确该接口是否应匿名访问；若应公开，放行安全白名单；若不公开，小程序端应改用专用公开配置接口 |
| QA-02 | P1 | 本地微信登录存在 `invalid appid` 风险 | 用户身份、会员权益、订单、支付链路无法完整验证 | 使用真实 AppID 或测试 AppID 配置，补齐登录链路验收 |
| QA-03 | P2 | 内容、会员 tabbar 当前都指向 `/pages/index/index` | 用户点击底部导航可能停留在首页，体验上像“点了没反应” | 发布配置中为不同导航配置真实页面路径，或在后台给出路径重复提示 |
| QA-04 | P2 | 小程序截图自动化曾出现 hang 风险 | 视觉回归依赖人工截图，无法稳定做像素级对比 | 保留 WXML/运行态验证，同时增加超时保护的截图脚本 |
| QA-05 | P3 | 版本列表字段 `releaseMode/createdAt/updatedAt` 在接口输出中为空 | 运营追溯发布原因、时间和发布模式不够直观 | 后端补齐字段或前端展示统一兜底说明 |

### 3. 测试经理结论

从质量角度看，当前“后台发布后小程序首页不同步”的主问题已经被修复到可验收观察状态；但还需要把公开配置权限和微信真实身份链路补齐，否则只能判定为“首页展示链路有条件通过”，不能判定整个产品完整闭环通过。

## 四、运营人员视角

### 1. 运营体验亮点

1. 版本管理有明确的发布版本记录，最新版本 `1.2.0` 可被接口识别为当前已发布版本。
2. 页面搭建结果能通过发布快照进入小程序运行态，降低“后台看到一个样，小程序看到另一个样”的运营风险。
3. 首页运营内容已经能在小程序端展示，包括活动 banner、快捷导航、会员卡片、文章列表和商品列表。
4. 版本历史保留了多个版本，具备后续回滚和发布追溯的基础。

### 2. 运营体验问题

1. 发布状态对运营来说还不够“所见即所得”。后台版本卡片显示已发布，但仍需要进入开发者工具或运行脚本才能确认小程序最终渲染结果。
2. 公开配置 403 会造成理解成本：运营人员看到后台配置已保存，但不知道小程序端是否真正使用的是当前配置、发布快照配置，还是兜底配置。
3. 版本字段缺少发布模式和更新时间，不利于运营人员判断“这是刚发布的版本还是历史版本”。
4. 导航路径重复没有明显提示，运营很可能配置出多个入口都回到首页的结果。

### 3. 运营建议

1. 在“小程序搭建”页面增加“当前线上生效预览”，直接展示线上发布快照，而不是只展示编辑态。
2. 发布前增加差异确认：显示本次发布相对当前线上版本改了哪些页面、组件、tabbar 和系统配置。
3. 对导航路径重复、页面不存在、组件类型不支持、数据源类型不支持做发布前校验，并给出阻断或警告。
4. 在版本卡片中展示发布时间、发布人、发布模式、发布备注，方便运营回溯。
5. 增加“一键用开发者工具预览/生成预览二维码”的运营入口，减少人工联调成本。

### 4. 运营视角结论

当前系统已经具备“搭建 -> 发布 -> 小程序首页生效”的基础运营能力，适合继续进行内容配置和演示；但若要用于正式日常运营，需要补齐发布前校验、线上预览和配置快照说明，否则运营人员仍容易误判当前线上真实效果。

## 五、用户视角

### 1. 用户实际看到的内容

通过微信开发者工具 WXML 检查，小程序首页能渲染以下内容：

1. 活动内容：“五一活动限时优惠”“点击了解”。
2. 底部导航：“首页”“内容”“会员”“我的”。
3. 会员信息：“金卡会员”“1280 积分”“5600 成长值”“3 优惠券”。
4. 内容入口与文章/商品相关组件已经存在。

这说明用户端不再是大量 `??` 或未知组件占位的状态，首页基础内容可以被正常消费。

### 2. 用户体验亮点

1. 首页结构比较完整，包含活动、导航、会员、内容、商品等常见运营模块。
2. 页面运行态无 loading 卡死，也没有返回 `error` 文案。
3. 底部导航可见，用户能理解主要入口。

### 3. 用户体验问题

1. 如果“内容”“会员”入口点击后仍停留在首页，用户会认为功能不可用或点击无反馈。
2. 登录失败会直接影响会员信息、优惠券、订单、支付等个性化功能，真实用户流程还没完全证明可用。
3. 领券、商品详情、预约、报名等操作未做真实闭环验证，当前只能确认首页展示，不代表交易和表单链路可用。

### 4. 用户视角结论

作为普通用户，当前首页已经能形成第一屏可用体验，内容比之前开发者工具截图中的异常态明显正常；但用户能否完成登录、领券、购买、预约、查看会员权益，还需要在真实 AppID 和正式接口权限配置下继续验证。

## 六、与用户最初问题的对应结论

用户最初反馈的问题是：后台已上线模块与开发者工具编译页面不一致、页面没有同步。

当前复测结论：

1. **页面 DSL 已同步**：公开首页接口返回的组件与微信开发者工具运行态一致，均为 `banner, nav, member_card, article_list, product_list`。
2. **未知组件问题已改善**：当前 WXML 未再出现 `未知组件: category_nav`，首页实际渲染了内容和会员信息。
3. **乱码/问号问题已改善**：WXML 文本样本中出现正常中文内容，不再是整屏 `??`。
4. **仍有配置风险**：公开配置接口 403，需要继续处理，否则后续主题、tabbar 或全局设置仍可能在某些环境不同步。

## 七、最终验收结论

| 维度 | 结论 | 说明 |
|---|---|---|
| 后台基础可用性 | 通过 | 管理员登录、版本列表、页面列表可访问 |
| 小程序首页同步 | 通过 | 后台发布 DSL 与小程序运行态组件一致 |
| 小程序首页渲染 | 通过 | WXML 有实际运营文案，页面非 loading、无 error |
| 运营发布闭环 | 有条件通过 | 已具备发布快照能力，但缺少可视化线上预览和发布前校验 |
| 用户完整业务闭环 | 未完全通过 | 登录、支付、预约、领券等真实链路未验证 |
| 配置公开能力 | 未通过 | `/api/v1/mp/config/public` 返回 403 |

**总判定**：有条件通过。  
**建议上线策略**：可以作为演示版或内部联调版继续使用；若要作为正式小程序运营系统交付，需要优先修复公开配置接口权限、真实微信登录 AppID 配置、导航路径重复提示，并补齐用户核心交易/报名/预约链路测试。

---

## 八、下一步整改优先级

1. P1：确认并修复 `/api/v1/mp/config/public` 的访问权限或接口设计。
2. P1：配置有效微信 AppID，完成 `wx.login -> 后端登录 -> 用户态数据` 验证。
3. P2：发布前校验导航路径和未知组件，避免上线后出现“点了没反应”或组件占位。
4. P2：增加稳定的自动截图/视觉回归脚本，用于对比后台预览与开发者工具渲染结果。
5. P3：补齐版本管理字段展示，包括发布时间、发布人、发布模式和备注。
