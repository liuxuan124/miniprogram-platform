## 1. 订单状态机

### 1.1 订单状态枚举

| 状态值 | 状态名 | 说明 |
| --- | --- | --- |
| 0 | 待支付 | 订单已创建，等待用户支付 |
| 10 | 待发货 | 实物商品已支付，等待发货 |
| 11 | 待核销 | 数字商品已支付，等待核销 |
| 12 | 待确认 | 服务商品已支付，等待服务确认 |
| 20 | 已发货 | 实物商品已发货，等待用户签收 |
| 30 | 已完成 | 订单已完成（签收/核销/服务确认） |
| 40 | 已取消 | 用户主动取消或超时未支付自动取消 |

> 状态值预留间隔，便于后续扩展（如 21=部分发货、31=部分完成等）

### 1.2 退款状态枚举（独立于订单状态）

| 状态值 | 状态名 | 说明 |
| --- | --- | --- |
| 0 | 待审核 | 用户申请退款，等待管理员审核 |
| 1 | 审核通过 | 管理员批准退款 |
| 2 | 审核拒绝 | 管理员拒绝退款 |
| 3 | 退款中 | 已向微信发起退款，等待回调 |
| 4 | 已退款 | 退款成功，资金已退回 |
| 5 | 退款失败 | 微信退款接口返回失败 |

### 1.3 支付状态枚举

| 状态值 | 状态名 | 说明 |
| --- | --- | --- |
| 0 | 待支付 | 等待用户支付 |
| 1 | 支付成功 | 微信回调确认支付成功 |
| 2 | 支付失败 | 支付失败或用户取消 |
| 3 | 已关闭 | 超时未支付，微信关闭订单 |

### 1.4 状态流转图

```
[创建订单] → 待支付(0)
                │
                ├── 用户支付成功 → 根据商品类型：
                │       ├── 实物商品 → 待发货(10)
                │       ├── 数字商品 → 待核销(11)
                │       └── 服务商品 → 待确认(12)
                │
                ├── 用户取消 → 已取消(40)
                │
                └── 超时未支付 → 已取消(40)

待发货(10) → 填写物流单号发货 → 已发货(20)
已发货(20) → 用户签收/自动确认 → 已完成(30)

待核销(11) → 管理员核销 → 已完成(30)

待确认(12) → 管理员确认服务 → 已完成(30)

已完成(30) ← 不可逆转

已取消(40) ← 不可逆转
```

### 1.5 状态流转规则

| 当前状态 | 允许操作 | 目标状态 | 触发方 | 条件 |
| --- | --- | --- | --- | --- |
| 待支付(0) | 支付成功 | 待发货(10)/待核销(11)/待确认(12) | 支付回调 | 实物/数字/服务 |
| 待支付(0) | 用户取消 | 已取消(40) | 小程序用户 | 仅限未支付订单 |
| 待支付(0) | 超时取消 | 已取消(40) | 系统定时任务 | 超过支付超时时间（默认30分钟） |
| 待发货(10) | 发货 | 已发货(20) | 管理员 | 填写物流公司和单号 |
| 待发货(10) | 申请退款 | 退款待审核 | 小程序用户 | 创建退款记录 |
| 已发货(20) | 确认收货 | 已完成(30) | 小程序用户 | 用户手动确认或自动确认（发货15天） |
| 已发货(20) | 申请退款 | 退款待审核 | 小程序用户 | 创建退款记录 |
| 待核销(11) | 核销 | 已完成(30) | 管理员 | 核销码验证通过 |
| 待核销(11) | 申请退款 | 退款待审核 | 小程序用户 | 创建退款记录 |
| 待确认(12) | 服务确认 | 已完成(30) | 管理员 | — |
| 待确认(12) | 申请退款 | 退款待审核 | 小程序用户 | 创建退款记录 |
| 已完成(30) | 申请退款 | 退款待审核 | 小程序用户 | 在售后时效内（默认7天） |

**禁止操作**：
- 已取消(40) 和 已完成(30) 状态下不可再变更订单状态（退款走独立流程）
- 待支付(0) 不可发货/核销/确认
- 已发货(20) 不可再发货

## 2. 退款流程

### 2.1 退款流程图

```
[用户申请退款] → 创建退款记录(待审核)
                      │
                      ├── 管理员审核通过 → 退款状态=审核通过
                      │       │
                      │       └── 调用微信退款API → 退款状态=退款中
                      │               │
                      │               ├── 微信退款回调成功 → 退款状态=已退款
                      │               │       │
                      │               │       └── 更新订单状态（全额退款→已取消，部分退款→保持原状态）
                      │               │
                      │               └── 微信退款回调失败 → 退款状态=退款失败
                      │                       │
                      │                       └── 记录失败原因，人工处理
                      │
                      └── 管理员审核拒绝 → 退款状态=审核拒绝
                              │
                              └── 通知用户退款被拒绝
```

### 2.2 退款规则

| 规则 | 说明 |
| --- | --- |
| 退款申请条件 | 订单状态为待发货/待核销/待确认/已发货/已完成，且在售后时效内 |
| 全额退款 | 退款金额 = 订单实付金额，退款成功后订单状态更新为已取消 |
| 部分退款 | 退款金额 < 订单实付金额，退款成功后订单状态保持不变 |
| 多次退款 | 单笔订单支持多次部分退款，累计退款金额不超过实付金额 |
| 退款审核 | 所有退款必须经过管理员审核，防止误操作 |
| 退款资金来源 | 默认从商户可用余额退款，余额不足时退款失败 |
| 退款时效 | 微信退款接口通常 1~3 个工作日到账 |
| 优惠券返还 | 全额退款时返还已使用的优惠券；部分退款时不返还 |

### 2.3 退款金额计算

```
退款金额 = 申请退款金额
约束条件：
  1. 退款金额 > 0
  2. 退款金额 ≤ 订单实付金额 - 已退款总额
  3. 退款金额精度为分（整数分，展示为元）
```

## 3. 微信支付规则

### 3.1 统一下单流程

```
1. 用户在小程序确认订单 → POST /api/v1/mp/orders
2. 后端创建订单，状态=待支付
3. 用户发起支付 → POST /api/v1/mp/orders/{id}/pay
4. 后端调用微信支付统一下单API（JSAPI）
   - 生成 out_trade_no（商户订单号）
   - 记录幂等键
5. 微信返回 prepay_id
6. 后端生成支付签名参数返回小程序
7. 小程序调用 wx.requestPayment 拉起支付
```

### 3.2 支付参数生成

| 参数 | 说明 | 来源 |
| --- | --- | --- |
| appId | 小程序 AppID | 系统配置 |
| timeStamp | 时间戳 | 当前时间 |
| nonceStr | 随机字符串 | UUID |
| package | 格式：prepay_id=xxx | 统一下单返回 |
| signType | 签名类型 | RSA（APIv3） |
| paySign | 签名 | 使用商户私钥签名 |

### 3.3 支付回调处理

```
1. 接收微信支付通知 → POST /api/v1/payment/wechat/notify
2. 验证回调签名（使用平台证书）
3. 解密回调数据（使用 APIv3 密钥）
4. 幂等检查：根据 out_trade_no 查询支付流水
   - 已处理：直接返回成功
   - 未处理：继续处理
5. 校验金额一致性：回调金额与订单金额一致
6. 更新支付流水状态=支付成功
7. 更新订单状态（根据商品类型）
8. 记录交易日志
9. 返回 200 OK 给微信
```

### 3.4 支付超时处理

| 规则 | 说明 |
| --- | --- |
| 支付超时时间 | 默认 30 分钟 |
| 超时处理 | 系统定时任务扫描超时未支付订单，调用微信关闭订单接口 |
| 关闭后状态 | 订单状态更新为已取消，取消原因为"超时未支付自动取消" |
| 并发处理 | 用户在超时边界支付成功时，以回调结果为准 |

## 4. 微信退款规则

### 4.1 退款接口调用

```
1. 管理员审核通过退款申请
2. 后端调用微信退款API
   - 生成 out_refund_no（商户退款单号）
   - 记录幂等键
3. 微信返回退款单号 refund_id
4. 退款状态更新为"退款中"
5. 等待微信退款回调
```

### 4.2 退款参数

| 参数 | 说明 | 来源 |
| --- | --- | --- |
| out_trade_no | 商户订单号 | 原支付流水 |
| out_refund_no | 商户退款单号 | 系统生成，唯一 |
| amount.refund | 退款金额（分） | 退款申请 |
| amount.total | 原订单金额（分） | 原支付流水 |
| amount.currency | 货币类型 | CNY |

### 4.3 退款回调处理

```
1. 接收微信退款通知 → POST /api/v1/payment/wechat/refund-notify
2. 验证回调签名
3. 解密回调数据
4. 幂等检查：根据 out_refund_no 查询退款流水
   - 已处理：直接返回成功
   - 未处理：继续处理
5. 根据退款结果更新退款状态
   - SUCCESS → 已退款
   - FAIL → 退款失败
   - CHANGE → 退款异常，人工处理
6. 全额退款成功时更新订单状态为已取消
7. 记录退款日志
8. 返回 200 OK 给微信
```

## 5. 幂等处理规则

### 5.1 支付幂等

| 场景 | 幂等策略 |
| --- | --- |
| 支付回调重复通知 | 根据 out_trade_no 查询支付流水，已处理则直接返回成功 |
| 用户重复点击支付 | 同一订单同一幂等键返回相同支付参数，不重复下单 |
| 并发支付 | 同一订单只允许一笔有效支付，使用数据库行锁 |

### 5.2 退款幂等

| 场景 | 幂等策略 |
| --- | --- |
| 退款回调重复通知 | 根据 out_refund_no 查询退款流水，已处理则直接返回成功 |
| 管理员重复审核 | 退款状态非"待审核"时拒绝操作 |
| 并发退款 | 同一退款单使用数据库行锁 |

### 5.3 幂等键生成规则

| 业务 | 幂等键格式 | 示例 |
| --- | --- | --- |
| 支付下单 | `pay:{order_id}:{timestamp}` | pay:123456:1715400000 |
| 支付回调 | out_trade_no（微信提供） | — |
| 退款申请 | `refund:{order_id}:{refund_count+1}` | refund:123456:1 |
| 退款回调 | out_refund_no（微信提供） | — |

## 6. 交易日志规则

### 6.1 日志记录要求

| 事件 | 必须记录 | 日志表 |
| --- | --- | --- |
| 统一下单 | out_trade_no、订单ID、金额、幂等键 | mp_payment_log |
| 支付成功 | transaction_id、回调原始数据、支付时间 | mp_payment_log |
| 支付失败 | 失败原因、回调原始数据 | mp_payment_log |
| 支付关闭 | 关闭原因、关闭时间 | mp_payment_log |
| 退款申请 | out_refund_no、退款ID、金额、幂等键 | mp_refund_log |
| 退款成功 | refund_transaction_id、回调原始数据 | mp_refund_log |
| 退款失败 | 失败原因、回调原始数据 | mp_refund_log |
| 订单状态变更 | 原状态、新状态、操作人、操作时间 | mp_operation_log |

### 6.2 日志安全规则

| 规则 | 说明 |
| --- | --- |
| 敏感信息脱敏 | 日志中不记录完整 API Key、证书私钥、用户银行卡号 |
| 回调数据保留 | 微信回调原始数据完整保留（JSON），用于问题排查 |
| 日志不可篡改 | 交易日志表禁止 UPDATE 和 DELETE 操作 |
| 日志保留周期 | 支付和退款日志至少保留 3 年 |

## 7. 并发与一致性规则

### 7.1 库存扣减

| 规则 | 说明 |
| --- | --- |
| 扣减时机 | 支付成功后扣减库存（非下单时） |
| 扣减方式 | 使用数据库乐观锁或 Redis 原子操作 |
| 库存不足 | 支付成功但库存不足时，自动退款并通知用户 |

### 7.2 金额一致性

| 规则 | 说明 |
| --- | --- |
| 支付金额校验 | 回调金额必须与订单 pay_amount 一致，否则拒绝处理 |
| 退款金额校验 | 退款金额 + 已退款总额 ≤ 订单 pay_amount |
| 金额精度 | 内部使用分（整数）计算，展示使用元（2位小数） |
| 金额存储 | 数据库使用 DECIMAL(10,2)，传输使用字符串 |

### 7.3 订单号生成

| 规则 | 说明 |
| --- | --- |
| 格式 | `MP` + 年月日时分秒(14位) + 6位随机数 |
| 示例 | MP202605111430000123456 |
| 唯一性 | 数据库唯一索引保证 |
| 不可猜测 | 随机数部分使用安全随机数生成器 |

## 8. 异常处理规则

| 异常场景 | 处理策略 |
| --- | --- |
| 支付回调到达但订单不存在 | 记录异常日志，调用微信关闭订单，原路退款 |
| 支付回调金额不一致 | 记录异常日志，订单标记为异常，人工处理 |
| 退款回调超时未到达 | 定时任务查询退款状态，超过24小时未回调则主动查询微信 |
| 微信接口调用失败 | 重试3次（间隔1s/2s/4s），仍失败则记录异常，人工处理 |
| 退款时商户余额不足 | 退款状态标记为退款失败，通知管理员充值后重试 |
| 并发支付同一订单 | 第一笔成功处理，后续回调幂等返回成功 |
| 订单超时但用户已支付 | 以支付回调结果为准，超时取消任务跳过该订单 |
